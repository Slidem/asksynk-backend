## Project overview

Asksynk is a productivity app, that's aimed to give user's full control over their attention, their schedule, and how / when they decided to respond to incoming messages from various communication channels.
By leveraging tags as a barrier between incoming communication channels and a user's schedule, users choose how tagged information will be answered. Incoming communication channels: in app message threads, integration with gmail, slack, whatsapp. Calendar integration: two way sync with different calendars (gcal), in app custom calendar. Asksynk will also provide public links for readonly views of a user's schedule (with possibility of directly addressing an issue on a user's timeblock) and also provides the ability to create "networks" of users, that have more access to someones calendar in their network (suggest timeblocks, suggest tasks etc).

## Task workflow

This repo uses **taskflow** (a local CLI) as the single enforcement point for a
file-based, parallel task workflow. All structural changes go through `taskflow`;
never hand-edit frontmatter or the tracker.

**Terminology**

- **task** (`T-NNN`) — a top-level unit of work.
- **sub-task** (`ST-NNN`, globally unique) — an independently buildable unit; runs in parallel.
- **note** — the author's free-form high-level thinking (one or more per task).

**Layout** (`tasks/`)

```
tasks/
  TASKS.md                 # tracker — GENERATED by `taskflow regen`, never hand-edited
  <task-slug>/
    task.md                # thin metadata (id, slug, title)
    notes/                 # author's high-level thinking
    subtasks/              # the generated plan; ST-NNN-<slug>.md, each carries source: <note-id>
```

**Rules**

- Sub-tasks within a task have **non-overlapping `owns:`** paths — that is what lets them
  run simultaneously. One git branch + worktree per sub-task.
- A sub-task is dispatchable when every id in its `depends_on` is `done`.
- `tasks/TASKS.md` is written **only** by `taskflow regen`.
- `taskflow validate` exits non-zero on structural errors — wire it into a pre-commit hook
  to gate commits, e.g. add to `.git/hooks/pre-commit`:
  ```sh
  #!/bin/sh
  taskflow validate || exit 1
  ```

**Commands**: `init` · `claude` · `new` · `note` · `subtask` · `status` · `validate` ·
`check` · `regen` · `next`. Skills `tf-plan` and `tf-work` and the `tf-task-implementer`
agent wrap these.
